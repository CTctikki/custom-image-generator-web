# 电商生图功能设计

## 目标

新增一个顶部主功能 Tab：`电商生图`。用户上传一张商品底图并输入商品标题后，系统一键生成电商文案和 4 张正方形商品图：主图、场景图、卖点图、白底图。

第一版目标是打通本地可测试闭环：

- 文案生成：调用 OpenAI-compatible `/v1/chat/completions`。
- 图片生成：调用 OpenAI Images edits，以商品底图作为参考图。
- 模块内展示本次结果，并把成功图片写入现有历史记录。
- 不接 COS 存储桶和数据库，但结果结构为后续扩展保留空间。

## 范围

第一版包含：

- 顶部第三个 Tab：`电商生图`。
- 商品标题输入。
- 商品底图上传。
- 文本模型输入框，默认 `gpt-5.5`。
- 图片模型输入框，默认 `gpt-image-2`。
- 复用现有全局 `Base URL` 和 `API Key`。
- 一键生成：先生成文案，再并发生成 4 张图。
- 文案结果可编辑：3 个卖点、长标题、短标题。
- 使用当前文案重新生成 4 张图片。
- 4 张结果图：主图、场景图、卖点图、白底图。
- 成功图片写入现有历史记录。

第一版不包含：

- COS 存储桶上传。
- 数据库存储。
- SKU 查询。
- 批量商品生成。
- 3 张单卖点图轮播。
- 独立的电商 Base URL / API Key 配置。

## 架构

采用“独立 Tab + 独立业务模块”的方式。

`App.tsx` 负责：

- 顶部 Tab 切换。
- 电商生图页面状态。
- 商品图上传和预览。
- 文案编辑区渲染。
- 4 张图片结果卡渲染。
- 把成功结果写入现有 `history`。

新增 `src/ecommerceGeneration.ts` 负责：

- 电商文案 prompt 构造。
- 电商图片 prompt 构造。
- 文本 API 请求。
- 文案 JSON 解析与校验。
- 4 类图片任务定义。
- 单张电商图片生成请求。

这样可以让第一版快速接入现有界面，同时避免继续把业务编排全部塞进 `App.tsx`。

## API 配置

电商模块复用现有 `workspace.baseUrl` 和 `workspace.apiKey`。

文本 API：

- 协议：OpenAI-compatible chat completions。
- 请求路径：`/v1/chat/completions`。
- 默认模型：`gpt-5.5`，用户可在电商模块里编辑。
- 输出：严格 JSON。

图片 API：

- 协议：OpenAI Images edits。
- 请求路径：`/v1/images/edits`。
- 默认模型：`gpt-image-2`，用户可在电商模块里编辑。
- 输入：商品底图作为 `image`。
- 尺寸：默认 `2K` 正方形，对 OpenAI Images 映射为 `1024x1024`。
- `output_format`：`png`。

如果全局 `Base URL` 或 `API Key` 缺失，电商模块提示用户先回 Image Studio 配置。

## 数据流

首次一键生成：

1. 用户上传商品底图。
2. 用户填写商品标题。
3. 用户确认或修改文本模型、图片模型。
4. 点击 `一键生成`。
5. 调用文本 API 生成：
   - `selling_points[0]`
   - `selling_points[1]`
   - `selling_points[2]`
   - `long_title`
   - `short_title`
6. 文案结果填入可编辑区。
7. 使用文案和商品底图并发生成 4 张图：
   - 主图
   - 场景图
   - 卖点图
   - 白底图
8. 4 张图在电商模块结果区展示。
9. 每张成功图写入现有历史记录。

重新生成图片：

1. 用户编辑卖点、长标题或短标题。
2. 点击 `重新生成图片`。
3. 跳过文本 API。
4. 使用当前文案并发生成 4 张图。
5. 新结果更新模块内展示，并写入历史记录。

重新生成文案：

1. 用户点击 `重新生成文案`。
2. 只刷新文案编辑区。
3. 不自动覆盖已有图片结果。

## UI 结构

`电商生图` 页面沿用线上现有工作台风格，不做独立品牌化大改。

页面分为三块：

### 商品输入面板

包含：

- 商品底图上传区。
- 商品标题输入框。
- 文本模型输入框，默认 `gpt-5.5`。
- 图片模型输入框，默认 `gpt-image-2`。
- 配置提示：显示当前使用全局 `Base URL / API Key`。

### AI 文案面板

包含可编辑字段：

- 长标题。
- 短标题。
- 卖点 1。
- 卖点 2。
- 卖点 3。

文案生成前为空。首次一键生成后自动填入。用户可编辑后重新生成图片。

### 图片结果面板

固定 4 个结果卡：

- 主图。
- 场景图。
- 卖点图。
- 白底图。

每个结果卡支持状态：

- 待生成。
- 生成中。
- 成功。
- 失败。

成功时显示图片和下载按钮。失败时显示完整错误文案。

## 生成规则

所有图片通用规则：

- 商品底图是商品外观唯一依据。
- 严格保留商品颜色、版型、轮廓、材质、图案、结构、细节、配件、比例和整体设计。
- 如果底图是随手拍，移除杂乱背景、手部、桌面、墙面、反光、阴影杂质等无关元素。
- 输出必须为 1:1 正方形。
- 图片风格干净、高级、清晰，商品主体突出。
- 不添加水印、无关 logo、乱码、无关品牌元素。
- 商品完整展示，不裁切关键部位，不让主体过小。

主图：

- 作为电商链接第一张图。
- 允许加入简洁醒目的卖点文字。
- 卖点文字优先使用 3 个 AI 卖点。
- 左下角约 20%-25% 区域必须保持干净留白。
- 左下角不能有文字、标签、icon、卖点说明、装饰元素或重要主体内容。
- 文案建议放在上方、右侧或右上区域。

场景图：

- 根据商品标题和卖点自动推断最合适使用场景。
- 场景真实、自然、高级，并强化商品卖点。
- 商品仍为画面主体。
- 图片中不能出现任何文字、标题、卖点文案、logo、水印或标签。

卖点图：

- 第一版生成 1 张综合卖点图。
- 图片中必须包含清晰、有吸引力的卖点文字。
- 文字优先使用 3 个 AI 卖点。
- 可以结合局部特写、细节放大、功能示意或近景展示。
- 文案不能遮挡商品关键部位。

白底图：

- 生成标准电商白底图。
- 背景必须纯白。
- 商品主体居中、完整、清晰。
- 可保留自然轻微接地阴影。
- 不出现文字、logo、水印、标签、边框、贴纸、人物或其他无关物品。

## 历史记录

4 张成功图都会写入现有历史记录。

历史记录字段映射：

- `imageDataUrl`：生成图片 Data URL。
- `mimeType`：返回图片 MIME type。
- `prompt`：带类型前缀的完整图片生成 prompt，例如 `[电商主图] 商品标题...`。
- `modelName`：电商图片模型。
- `protocol`：`openai_images`。
- `aspectRatio`：`1:1`。
- `imageSize`：`2K`。
- `inputImageNames`：商品底图文件名。
- `createdAt`：生成完成时间。

第一版不改 `HistoryItem` schema。后续接 COS 和数据库时，可在新存储层增加：

- `batchId`
- `assetType`
- `productTitle`
- `sellingPoints`
- `longTitle`
- `shortTitle`
- `remoteUrl`
- `storageProvider`

## 错误处理

文案生成失败：

- 停止后续图片生成。
- 在状态区显示用户可读错误。
- 文案区保留当前已有内容。

文案 JSON 解析失败：

- 尝试从响应中提取第一个 JSON 对象。
- 校验 `selling_points` 至少 3 项，`long_title` 和 `short_title` 为非空字符串。
- 校验失败则显示“文案返回格式不正确，请重新生成文案”。

图片生成失败：

- 单张失败不影响其他图片。
- 对应结果卡进入失败状态并显示完整错误。
- 成功的图片仍写入历史记录。

配置缺失：

- 缺少 `Base URL`、`API Key`、商品标题、商品底图、文本模型或图片模型时，生成按钮不可用或给出明确提示。

## 测试与验收

新增或扩展 contract test 检查：

- 顶部存在 `电商生图` Tab。
- 默认文本模型为 `gpt-5.5`。
- 默认图片模型为 `gpt-image-2`。
- 电商文案 prompt 要求 JSON 输出，并包含 `selling_points`、`long_title`、`short_title`。
- 主图 prompt 包含左下角 20%-25% 留白要求。
- 场景图 prompt 禁止出现文字。
- 卖点图 prompt 要求包含卖点文字。
- 白底图 prompt 要求纯白背景。
- 所有图片 prompt 要求 1:1 正方形。
- 电商图片生成使用 OpenAI Images edits。
- 电商图片默认尺寸为 `1024x1024`。

验证命令：

- `npm test`
- `npm run build:web`

本地视觉验证：

- 启动 `npm run dev`。
- 打开本地页面。
- 确认 `电商生图` Tab 可访问。
- 检查移动端和桌面端布局不重叠。
- 使用测试商品标题和商品图验证状态流转、失败提示、结果展示和下载按钮。
